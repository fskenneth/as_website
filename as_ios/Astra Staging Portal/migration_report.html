1. Executive Summary — Hybrid-First Strategy

Core Principle: Don't rebuild what works. Build what Zoho can't do.
Keep Zoho Creator for battle-tested internal operations (time tracking, scanning, expenses, packing). Build the website for what it does better (customer experience, 3D design, instant quote, Stripe payments, SEO). Selectively migrate features from Zoho to the website only when there is a clear reason to — not on a predetermined schedule.

The Staging Manager Zoho Creator application is a comprehensive operations platform with 50+ forms, 47 reports, 32+ pages, 106 custom functions, and 6 scheduled tasks. Rebuilding all of this from scratch while running a business is high-risk and unnecessary. Instead, the hybrid-first approach splits ownership between the two systems based on what each does best.

The Sync Layer Is the Bridge
The existing Playwright-based Zoho sync (30-second interval, 0 API calls) + write-back queue is the critical bridge between systems. Keep it running and robust. It lets the website read inventory data and push confirmed stagings back to Zoho. This bridge enables the hybrid approach and can be removed only when a feature is fully migrated.

6. Workflows & Automations

Zoho Creator workflows are event-driven automations that fire on record creation, editing, deletion, or field changes. These contain the core business logic of the application.

Staging Lifecycle Workflows

Quote → Staging Conversion (On Add Load)

When a new Staging record is created with a Quote reference:

  • Sets Staging_Status = "Inquired"
  • Copies from Quote: Customer_CRM, customer details, Staging_Term, Initial_Staging_Fee, Photography_Fee, Property_Type, Occupancy_Type, Staging_Item_List, Seller_To_Do_List
  • Links bidirectionally: Quote.Staging = new Staging ID
Staging Status Change → Item Movement (On User Input)

When Staging_Status changes to "Inactive":

  • Finds ALL items at the staging address
  • Moves each item back to storage via updateItemCurrentLocationByDirection(item.ID, "in", "Staging Inactive")
  • Updates item move counts and location history
Staging Deletion → Cascade Delete

When Delete_Staging is checked or staging record deleted:

  • Deletes all Property_Image records for this staging
  • Deletes all Item_Position records
  • Deletes all Area records
  • Deletes the Staging record
Finish Destaging → Return All Items to Storage

When Finish_Destaging is checked:

  • Finds all items currently at the staging address
  • Moves each back to storage with "Destaging Finish + timestamp"
  • Updates item location history
HST & Fee Calculation (On User Input of Pay_by_Cash)

Dynamic fee recalculation:

  • If Total_Staging_Fee < $3: No HST, Total = Staging Fee
  • If Pay_by_Cash: HST = 0
  • Otherwise: HST = Total_Staging_Fee × 0.13 (13%)
  • Total_Amount = Total_Staging_Fee + HST
  • Owing_Amount = Total_Amount - Paid_Amount (summed from all Payment records)

Area & Design Workflows

Area Creation → Auto-Generate Item Positions

When a new Area is added to a staging:

  • Calls createItemPositionList(area_id)
  • Generates placeholder Item_Position records based on area type (e.g., Living Room gets: Sofa, Coffee Table, 2x Accent Chair, End Table, Floor Lamp, etc.)
  • Each position has: Item name, area reference, staging reference
Area Name Change → Rebuild Positions

When Area_Name is changed on edit:

  • Deletes ALL existing Item_Position records for this area
  • Creates new positions based on the new area type
  • Updates all position display names

Barcode Scan Workflows

Barcode Scan → Item Location Update or Check

When barcode scanned (auto-submits):

  • 2-char barcode (Rack): Sets scan setting rack number, switches to storage location
  • BAG/BIN prefix: Displays container info (floor + area)
  • Item barcode: Based on Scan_Mode:
    • "Update Location": Updates Item.Current_Location, assigns rack, increments Move_Count, assigns to matching Item_Position if at a staging property
    • "Check Item": Opens item detail page for viewing
  • Deletes the scan record after processing (transient)

Invoice & Email Workflows

Invoice Auto-Population (On Add Load)

When creating a new Staging_Invoice:

  • Invoice_Date = today
  • Due_Date = Staging_Date (or today if past)
  • Staging_Fee = Initial_Staging_Fee
  • Charge_1: Extension fees (if > 0)
  • Charge_2: Photography fees (if > 0)
  • Charge_3: Moving fees (if > 0)
  • Paid_Amount, Pay_by_Cash from Staging
Email Generator → Send & Track (On Success)

When Email_Generator form submitted:

  • On Load: Fetches customer email from Zoho CRM (500/day limit), populates email template based on type
  • On Success: Sends email via sendmail with To/CC/BCC/attachments
  • If "Payment Notification": Sets Staging.Invoice_Sent_Date = today
  • Always BCCs sales@astrastaging.com
  • Updates staging table HTML for display

Customer Portal Workflows

Customer Portal → Dynamic Content Generation

The most complex workflow. Based on Portal_Type:

  • Staging: Shows terms, staging item list, rental period, extension offers, destaging info. Generates HTML email with portal button link.
  • Extension: Offers 50% discount on renewal. Portal button for customer to accept/decline.
  • Destaging: Requests destaging confirmation and access information. Marks as "*Action Required*" in email.

Portal URL format: https://creatorapp.zohopublic.com/astrastaging/staging-manager/form-perma/Astra_Staging_Customer_Portal/[PUBLIC_ID]?Staging=[id]&Portal_Type=[type]

RFID & Inventory Workflows

RFID Scan → Bulk Location Update + Email

When Inventory_Check record created:

  • Calls updateItemCurrentLocationRfidScan() to process all scanned RFID tags
  • Matches tags to items, updates locations
  • Sends email to scan@astrastaging.com with scan summary and link to Inventory_Update_Page

The Sync Layer (Critical Infrastructure)

The sync layer is the single most important piece of the hybrid architecture

The Playwright-based page sync (30-second interval) + write-back queue is what makes the hybrid possible. If it breaks, both systems are blind to each other's data. Invest in:

  • Health monitoring: Alert if sync hasn't run in 5 minutes
  • Auto-restart: Restart Playwright browser if it crashes
  • Conflict detection: Flag when both systems modify the same record
  • Audit log: Track every sync operation for debugging

Database Strategy

SQLite is fine for hybrid

With Zoho handling the heavy concurrent writes (employee operations), the website's SQLite only needs to handle customer traffic and read-only sync data. SQLite is more than sufficient. Re-evaluate for PostgreSQL only if Phase 4+ migrates high-write operations to the website.

Deployment Strategy: One Codebase, Two Deployments

Public site and webapp stay in one codebase — but run as two processes.

The FastHTML app serves both the public marketing site and the authenticated webapp (portal, design studio, admin, quote flow). They share auth, DB, models, and the Zoho sync bridge — splitting them into two repos would just create a sync-code-between-repos problem.

Isolation is achieved at the process layer, not the code layer:

  • astrastaging.com → uvicorn process A (public-facing, autoscaled for traffic spikes)
  • app.astrastaging.com → uvicorn process B (webapp routes, authenticated users)
  • api.astrastaging.com (optional) → uvicorn process C (mobile API, /api/v1/*)

All three run the same main.py. A webapp crash or memory leak doesn't take the marketing site down. Deployments can be rolled independently. The shared DB is still a shared dependency — mitigate with query timeouts and, if ever needed, a read replica.

Key Recommendations

1. Harden the Sync Layer First

Before building any new features, add health monitoring, auto-restart, and alerting to the Playwright sync. This is the foundation of the hybrid approach. If sync is fragile, everything else is at risk.

2. Customer-First, Operations-Later

All Phase 1-2 work is customer-facing. Zero employee disruption. This delivers business value immediately (better UX, more conversions) without touching the operations that already work.

3. API-First Everything

Every new feature on the website exposes a REST API endpoint. This ensures the mobile app (Phase 3) can immediately consume the same APIs without rebuilding.

4. No Forced Deadlines for Migration

The hybrid runs indefinitely. Zoho decommission is a natural consequence of all triggers being met, not a calendar date. Don't migrate for migration's sake.

5. Measure Before Migrating

Before moving any feature off Zoho: measure current usage (how often, by whom, on what device). After moving: compare. If the new version is worse, roll back. Keep Zoho as the safety net.

6. One Feature at a Time

Never migrate two operations features simultaneously. Move one, let it stabilize for 1+ month, collect feedback, then evaluate the next. This limits blast radius.

import SwiftUI

struct ContentView: View {
    // Simulator: use localhost. Physical iPhone: use "http://192.168.2.236:5001"
    private let apiBase = "http://localhost:5001"

    @State private var message = "Tap the button to call the server"
    @State private var serverTime = ""
    @State private var isLoading = false

    var body: some View {
        VStack(spacing: 24) {
            Text("Astra Hello").font(.largeTitle).bold()

            Text(message)
                .multilineTextAlignment(.center)
                .padding(.horizontal)

            if !serverTime.isEmpty {
                Text("Server time: \(serverTime)")
                    .font(.caption)
                    .foregroundStyle(.secondary)
            }

            Button(action: fetchHello) {
                HStack {
                    if isLoading { ProgressView() }
                    Text(isLoading ? "Loading..." : "Ping /api/v1/hello")
                }
                .padding(.horizontal, 24)
                .padding(.vertical, 12)
                .background(Color.blue)
                .foregroundStyle(.white)
                .clipShape(Capsule())
            }
            .disabled(isLoading)
        }
        .padding()
    }

    private func fetchHello() {
        guard let url = URL(string: "\(apiBase)/api/v1/hello") else { return }
        isLoading = true
        URLSession.shared.dataTask(with: url) { data, _, error in
            DispatchQueue.main.async {
                isLoading = false
                if let error = error {
                    message = "Error: \(error.localizedDescription)"
                    return
                }
                guard let data = data,
                      let obj = try? JSONSerialization.jsonObject(with: data) as? [String: Any] else {
                    message = "Invalid response"
                    return
                }
                message = (obj["message"] as? String) ?? "No message"
                serverTime = (obj["server_time"] as? String) ?? ""
            }
        }.resume()
    }
}

Tapping the button shows "Hello from as_website" with a UTC timestamp. This proves the FastHTML backend can serve as the iOS app's API server over LAN. All future /api/v1/* endpoints (auth tokens, staging reads, scan submissions, etc.) build on this same pattern.
